01 | 并发编程入门：一文吃透进程与线程，手写 4 种线程创建方式

很多 Java 工程师刚接触并发编程时，都会卡在「进程与线程」这第一步：

  • 背了一堆概念，合上书就分不清谁是资源分配单位，谁是调度单位；
  • 写线程代码只会抄模板，搞不懂start()和run()到底有啥区别；
  • 线程状态记了又忘，更别说理解状态之间的流转逻辑。

其实并发编程一点都不抽象，它本质就是解决「如何让多件事同时高效运行」的问题。今天咱们就从生活类比出发，配上可直接运行的代码和 ASCII 流程图，把并发编程的第一块基石砸实。

一、先搞懂：什么是进程？什么是线程？

1.1 生活类比，一秒理解

我们可以把操作系统比作一个产业园区，每个运行的程序就是园区里的一家「工厂」：

  • 进程：就是这家工厂本身，是操作系统分配资源（厂房、设备、原材料）的最小单位。每家工厂有独立的场地和资源，工厂之间互不干扰。

比如你电脑上同时开着微信、IDEA、浏览器，它们各自就是一个独立的进程。

  • 线程：就是工厂里的「工人」，是 CPU 调度执行的最小单位。一个工厂里至少有一个工人（主线程），也可以有多个工人同时干活，所有工人共享工厂的厂房、设备等资源。

比如微信里同时发消息、刷朋友圈、接收文件，背后就是多个线程在协同工作。

1.2 核心区别对比

维度 进程 线程
资源分配 操作系统分配资源的最小单位，拥有独立内存空间 CPU 调度执行的最小单位，不独立拥有资源，共享所属进程的资源
切换开销 开销大，需要切换整个内存空间、文件句柄等 开销小，只需切换线程的栈、寄存器等少量数据
通信成本 成本高，需要通过管道、消息队列、共享内存等方式 成本低，直接读写进程的共享内存即可
稳定性 进程之间相互隔离，一个进程崩溃不影响其他进程 线程共享进程资源，一个线程崩溃可能导致整个进程挂掉

1.3 为什么要用多线程，而不是多进程？

核心原因就是轻量、高效：

  • 线程的创建和销毁开销远小于进程；
  • 线程之间的上下文切换成本远低于进程；
  • 同进程内的线程共享内存，数据通信更简单高效。

当然多进程也有优势：隔离性强、稳定性高，比如浏览器的多进程架构，一个标签页崩溃不会拖垮整个浏览器。

二、Java 中创建线程的 4 种方式

Java 中线程的核心类是java.lang.Thread，它是对操作系统线程的封装。创建线程本质就是构造一个 Thread 对象，给它分配要执行的任务。

下面我们逐个讲解 4 种创建方式，每一种都附上可直接运行的完整代码。

方式 1：继承 Thread 类，重写 run () 方法

这是最直观的方式：自定义一个类继承 Thread，重写run()方法编写线程要执行的业务逻辑，然后创建子类对象，调用start()启动线程。

优缺点：

  • 优点：代码简单直观
  • 缺点：Java 单继承限制，无法再继承其他类；任务和线程绑定，不利于资源复用

方式 2：实现 Runnable 接口，重写 run () 方法

将任务逻辑抽离出来，实现 Runnable 接口，再将任务对象交给 Thread 去执行，实现了「线程对象」和「任务逻辑」的解耦。

优缺点：

  • 优点：突破单继承限制；任务与线程解耦，同一个任务可以交给多个线程执行；适合多个线程共享同一份资源的场景
  • 缺点：任务没有返回值，无法抛出受检异常

补充：日常开发中更常用匿名内部类或 Lambda 表达式简化写法，这是最主流的写法：

方式 3：Callable + FutureTask，带返回值的线程

Runnable 的run()没有返回值，如果我们需要线程执行完返回一个结果（比如异步计算），就需要用 Callable 接口。

Callable的call()方法可以返回结果，也可以抛出异常；配合FutureTask包装任务，交给 Thread 执行，最后通过get()方法获取结果。

优缺点：

  • 优点：支持获取任务返回值，支持抛出异常，功能更强大
  • 缺点：代码稍复杂；get()方法会阻塞主线程

方式 4：线程池创建线程

前面三种都是手动创建线程，在生产环境中，频繁创建销毁线程会浪费大量资源，因此我们通常用线程池来管理和复用线程。

核心结论：

注意：一个线程对象只能调用一次start()，多次调用会抛出IllegalThreadStateException异常。

三、线程的 6 种状态与完整生命周期

线程从创建到销毁，并不是一直处于运行状态，它会在不同状态之间流转。Java 中线程的状态定义在Thread.State枚举类中，一共 6 种。

3.2 逐个拆解 6 种状态

3.2.1 NEW（新建状态）

当我们new Thread()创建了线程对象，但还没调用start()方法时，线程就处于新建状态。

此时它只是一个 Java 对象，还没有和操作系统的真实线程绑定。

3.2.2 RUNNABLE（可运行状态）

调用start()之后，线程就进入可运行状态。Java 层面的 RUNNABLE 对应操作系统层面的两种情况：

  1. 就绪（Ready）：线程已经准备好，等待 CPU 调度执行
  2. 运行中（Running）：CPU 正在执行这个线程的代码

因为 Java 不区分就绪和运行中，统一称为可运行状态。

3.2.3 BLOCKED（阻塞状态）

线程在等待获取锁（synchronized 锁）时，会进入阻塞状态。

比如两个线程同时抢同一把锁，没抢到的线程就会进入 BLOCKED 状态，直到拿到锁才会回到 RUNNABLE。

3.2.4 WAITING（无限等待状态）

线程进入无时间限制的等待，直到被其他线程唤醒。触发场景：

  • Object.wait() 方法
  • Thread.join() 方法
  • LockSupport.park() 方法

3.2.5 TIMED_WAITING（计时等待状态）

线程进入有时间限制的等待，时间到了自动唤醒。触发场景：

  • Thread.sleep(long millis)
  • Object.wait(long timeout)
  • Thread.join(long millis)
  • LockSupport.parkNanos() / parkUntil()

3.2.6 TERMINATED（终止状态）

线程执行完run()方法正常结束，或者执行过程中抛出未捕获的异常，就会进入终止状态，生命周期结束。

五、本篇核心总结

  1. 进程是操作系统资源分配的最小单位，线程是 CPU 调度的最小单位；线程轻量、切换开销小，是并发编程的核心载体。
  2. Java 创建线程有 4 种方式：继承 Thread、实现 Runnable、Callable+FutureTask、线程池，日常开发优先使用 Runnable 配合线程池。
  3. start()才是真正启动线程的方法，run()只是普通方法调用。
  4. 线程有 6 种状态：NEW、RUNNABLE、BLOCKED、WAITING、TIMED_WAITING、TERMINATED，状态流转有明确的触发条件。
  5. 多线程访问共享资源时，会出现线程安全问题，这是后续学习的核心重点。

课后练习

为什么实际开发中更推荐使用 Runnable 接口，而不是继承 Thread 类？

推荐 Runnable 的原因：

  • 突破 Java 单继承的限制，类还可以继承其他父类。
  • 解耦：将任务逻辑与线程对象分离，同一个任务可以交给多个线程执行，便于资源共享。
  • 增强代码可复用性和可维护性，任务可以被线程池、其他组件复用。

一个线程对象可以多次调用start()方法吗？为什么？

不可以。

一个线程只能调用一次start()方法。Thread 对象内部有一个线程状态标记（threadStatus），调用 start () 后状态会从 NEW 变更，再次调用时会检测到状态已不是 NEW，从而抛出IllegalThreadStateException异常。这是因为操作系统层面的线程只能被创建一次。

调用 Thread 类的run()方法和start()方法有什么本质区别？

run()是普通成员方法，直接调用不会创建新线程，代码在当前线程中同步执行。

start()是线程启动方法，调用后线程进入就绪状态，由 JVM 底层创建操作系统线程，在新线程中异步执行run()方法。

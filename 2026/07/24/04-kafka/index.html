04 | Topic 与 Partition：Kafka 消息存储的底层逻辑

Topic 和分区到底是什么关系？消息真的存在 Topic 里吗？

为什么都说 “分区越多，吞吐量越高”？

既然 Kafka 保证消息顺序，为什么还会出现消息乱序？

这些问题的答案，全都藏在 Kafka 的存储设计里。今天我们就深入底层，把 Topic 和 Partition 彻底讲透，从逻辑概念到物理文件，从设计初衷到查找机制，全程用快递驿站类比 + 原理图，带你看懂 Kafka 高吞吐、高可靠的底层秘密。

一、先分清：Topic 是逻辑概念，Partition 是物理实体

很多新手最大的误区，就是以为 Topic 是真实存数据的地方。其实恰恰相反，Topic 只是一个逻辑分类标签，真正承载消息的是 Partition（分区）。

1.1 两者的层级关系

一个 Topic 包含一个或多个 Partition，分区数量可以在创建 Topic 时指定，后续也能增加。所有分区合在一起，才构成了这个 Topic 的完整消息数据。

1.2 分布式分布：分区分散在不同 Broker

更重要的一点是：同一个 Topic 的分区，并不是都放在同一个 Broker 上的，而是会均匀分散在集群的不同 Broker 节点上。

这是 Kafka 分布式架构的核心设计：数据分片存储，每台机器只存一部分，既突破了单机容量限制，又能让多台机器同时提供读写服务，整体吞吐量翻倍。

二、为什么要有分区？两大核心价值

分区不是为了故意增加概念复杂度，它是 Kafka 能支撑百万级 TPS 的核心设计，本质解决两个问题：容量扩展和并发读写。

2.1 水平扩展：突破单机容量上限

如果一个 Topic 所有消息都存在一台机器上，那这台机器的磁盘有多大，Topic 最多就能存多少数据。业务量增长、消息越积越多，单机磁盘迟早会满，而且没法扩容。

有了分区之后，数据被拆成多份，分散在不同 Broker 上。集群有多少台机器，就能存多少倍的数据，业务增长了，加机器、加分区就能扩容，理论上容量没有上限。

这就像一个仓库放不下所有货物，那就多建几个仓库，每个仓库存一部分，货再多也不怕。

2.2 并发读写：吞吐量的核心来源

这是分区最核心的价值，也是 “分区越多吞吐越高” 的根本原因。

我们先想一个问题：如果一个 Topic 只有 1 个分区，生产者发消息、消费者读消息，都只能操作这一个分区，同一时间只能有一个线程在读写，相当于单车道，再快也有限。

如果分成 3 个分区，就相当于把单车道扩成了三车道：

  • 写入端：生产者可以同时往 3 个分区发消息，3 个分区并行写入，写入速度翻 3 倍。
  • 消费端：消费者组里可以有 3 个消费者，同时读取 3 个分区，消费速度也翻 3 倍。

Kafka 的并发读写能力，是以分区为单位的。分区越多，能支持的并发数就越高，整体吞吐量也就越大。这也是为什么生产环境的 Topic，很少只设置 1 个分区。

三、关键特性：分区内严格有序，分区间无序

这是 Kafka 最核心、也是面试最高频的知识点，面试官可能会问：“如何保证Kafka中消息的顺序性”。很多人学了很久都没搞透。

3.1 为什么单分区能保证严格有序？

每个 Partition 在磁盘上对应一个日志文件，消息采用顺序追加写入的方式：新来的消息永远写在文件的最后面，不会插入到中间，也不会修改已有的消息。

就像我们往本子上写日记，永远从第一页往后写，每页按顺序编号，不会在中间插页。所以从前往后读的时候，消息的顺序和写入顺序是完全一致的。

划重点：同一个分区内的消息，严格按照写入顺序排列，Offset 单调递增，这是 Kafka 对顺序性的唯一保证。

3.2 为什么做不到全局有序？

既然单分区有序，那为什么不把所有消息都放一个分区，实现全局有序？

原因很简单：**全局有序和高吞吐是矛盾的，**两者不可兼得。

如果要整个 Topic 的消息全局有序，那就只能有 1 个分区，相当于单车道，所有消息都要排队一个个写、一个个读，吞吐量会非常低，完全发挥不出 Kafka 的性能优势。

而多分区场景下，每个分区各自独立写入，消息进入哪个分区是不确定的（轮询或哈希），最终不同分区之间的消息没有全局的先后顺序。

3.3 业务实践：如何实现 “业务级有序”

全局有序代价太大，那业务上需要保证顺序怎么办？比如同一个订单的创建、支付、发货消息，必须按顺序处理，不能乱。

答案是：不用全局有序，只保证同一个业务主键的消息有序即可。

做法也很简单：发送消息时，用业务主键（比如订单 ID）作为消息 Key。Kafka 默认的分区器会对 Key 做哈希计算，同一个 Key 的消息，一定会被路由到同一个分区里。这样同一个订单的所有消息（创建、支付、发货等）都在同一个分区，自然就保证了顺序。

四、深入底层：Partition 的日志分段存储

搞懂了分区的作用，我们再往深挖一层：一个分区就是一个日志文件吗？如果消息越来越多，文件越来越大，查找和清理岂不是很慢？

Kafka 早就想到了这个问题，它的解决方案是日志分段（Log Segment）：一个分区的日志不是一个大文件，而是分成很多个大小相等的小文件段，每个段就叫一个 Segment。

4.1 为什么要做分段？

如果只有一个日志文件，会有三个明显的问题：

  1. 文件太大，查找历史消息非常慢，要从头开始遍历。
  2. 删除过期消息很麻烦，不能直接删整个文件，只能截掉前面的部分。
  3. 大文件不利于操作系统的页缓存管理，性能会下降。

分成多个 Segment 之后，这些问题迎刃而解：

  1. 查找消息时，先定位到对应的 Segment，只需要在小文件里找，速度快很多。
  2. 删除过期消息时，直接删掉最旧的 Segment 文件即可，非常高效。
  3. 小文件更利于操作系统的 PageCache 缓存，提升读写性能。

用生活中的例子说：就像记账不会把所有账目都写在一本超级厚的账本里，而是按月分成一册一册的。找某一天的账，先找到对应的月份账本，再翻就快多了；过期的旧账本，直接整册归档或销毁，不用撕页。

4.2 Segment 的文件组成

每个 Segment 对应磁盘上的一组文件，核心有三个文件：

  • .log 文件：消息日志文件，真正存放消息内容的地方。
  • .index 文件：偏移量索引文件，记录消息 Offset 和物理位置的对应关系，用来快速定位消息。
  • .timeindex 文件：时间戳索引文件，记录时间戳和 Offset 的对应关系，用来按时间查找消息。

这三个文件是一一对应的，文件名就是该 Segment 第一条消息的起始 Offset，长度固定 20 位，不足补 0。比如第一个 Segment 从 0 开始，文件名就是 00000000000000000000.log。

小常识：一个分区同一时间只有一个活跃 Segment（Active Segment），也就是最新的那个段，所有新消息都写入这个活跃段。当活跃段大小达到阈值（默认 1GB）后，就会关闭，生成新的活跃段继续写入。

4.3 稀疏索引：又快又省空间的查找机制

有了索引文件，我们就能快速根据 Offset 找到对应的消息。但 Kafka 的索引不是每条消息都建一条索引项，而是采用稀疏索引的设计：每隔一定字节数，才记录一条索引。

为什么不做稠密索引？因为如果每条消息都建索引，索引文件本身会变得非常大，占用大量磁盘和内存，得不偿失。稀疏索引用极小的空间代价，换来了接近稠密索引的查找效率。

查找一条消息的完整过程是这样的：

  1. 定位 Segment：根据目标 Offset，通过二分查找找到它所在的 Segment 文件。
  2. 查索引文件：在对应 Segment 的.index 文件里，找到小于等于目标 Offset 的最大索引项，得到消息在.log 文件里的物理起始位置。
  3. 顺序扫描：从这个物理位置开始，顺序往后读，直到找到目标 Offset 的消息。

这个过程就像查书：先通过目录找到章节对应的页码，再从那一页开始往后翻，很快就能找到目标内容。

4.5 集群视角：分区与副本的分配规则

讲完了单个分区的存储，我们再回到集群视角，看看创建 Topic 时，分区和副本是怎么分配到各个 Broker 上的。

分配的两个核心原则

Kafka 分配分区副本时，遵循两个基本原则，目的是最大化可用性和负载均衡：

  • 均匀分布：所有分区的 Leader 副本尽量均匀分配到所有 Broker 上，保证每台机器的负载差不多。
  • 副本隔离：同一个分区的多个副本，绝对不能放在同一个 Broker 上。不然这台机器挂了，这个分区的所有副本都没了，数据就丢了。

实际分配示例

我们用最经典的场景举例：3 台 Broker（编号 0、1、2），创建一个 Topic，3 个分区，3 个副本（1 个 Leader + 2 个 Follower）。

  • 每个 Broker 都有一个分区的 Leader，读写请求由 Leader 处理，三台机器同时对外提供服务，性能最大化。

4.6 常见误区：分区数不是越多越好

很多新手会觉得：既然分区越多吞吐越高，那我直接设成 1000 个分区，岂不是性能拉满？

这是一个非常典型的误区。分区数确实和吞吐量正相关，但不是线性增长，而且分区过多会带来很多副作用。

4.6.1 分区过多的副作用

  1. 文件句柄耗尽：每个分区对应多个文件，分区越多，打开的文件句柄就越多，超过操作系统限制会报错。
  2. 客户端内存占用高：生产者和消费者会为每个分区维护缓冲区和请求队列，分区太多会占用大量内存。
  3. 故障恢复变慢：Broker 宕机时，上面的分区 Leader 需要重新选举，分区越多，选举和恢复的时间就越长。
  4. ISR 同步压力大：Controller 节点需要管理所有分区的状态，分区越多，管理开销越大，集群稳定性下降。

4.6.2 如何合理估算分区数

业界有一个通用的估算思路，可以作为参考：

  1. 先压测单分区的吞吐量，比如单分区写入能到 1 万条 / 秒，消费能到 2 万条 / 秒。
  2. 根据业务目标吞吐量计算：目标峰值写入吞吐 ÷ 单分区写入吞吐 ≈ 所需分区数。
  3. 再结合消费者并发需求，预留一定冗余，最终确定分区数。

举个例子：目标峰值写入是 5 万条 / 秒，单分区写入 1 万条 / 秒，那至少需要 5 个分区，再留一点冗余，设成 6-8 个就比较合理。

一般来说，单 Broker 上的分区数控制在几百个以内比较稳妥，不要盲目追求大数量。

六、写在最后

今天我们从逻辑概念到物理存储，从设计初衷到分配规则，把 Topic 和 Partition 彻底拆解了一遍，相信你已经明白 Kafka 为什么要这么设计了。

我们来总结一下核心要点：

  1. Topic 是逻辑分类，Partition 是物理存储单元，消息真正存在分区里，分区分散在不同 Broker 上实现分布式。
  2. 分区的核心价值是水平扩容和并发读写，是 Kafka 高吞吐的基石。
  3. 单分区内严格有序，分区间无序，业务上通过消息 Key 实现业务主键级别的顺序，兼顾性能与顺序性。
  4. 分区采用日志分段存储，配合稀疏索引，在节省空间的同时实现了消息的快速查找。
  5. 分区副本遵循均匀分布、副本隔离的原则，保证高可用和负载均衡，分区数不是越多越好，要合理设置。

七、课后疑问

一条消息会写入一个 topic 的多个分区吗？还是说一条消息只会写入到某一个分区？

结论：一条消息，只会被写入到同一个 Topic 的某一个分区里，绝对不会同时写入多个分区。

① 从设计本质看：分区是「分片」，不是「复制」

分区的核心作用是把数据拆分成多份、分散存储，实现并行读写，也就是典型的「数据分片」思路，目的是拆分压力、提升吞吐量。

如果一条消息同时放进多个分区，不仅成倍浪费存储空间，还彻底失去了分片的意义 —— 本来是为了拆分负载，结果数据重复存，反而更臃肿。

② 从生产者机制看：分区器只会选定一个目标分区

在生产者内部，消息发送前会经过「分区器（Partitioner）」，它的核心职责就是为每一条消息，选定且仅选定一个目标分区。

Kafka 默认的两种分区规则，最终结果都只会对应一个唯一的分区：

  1. 指定了消息 Key：对 Key 做哈希运算，再对分区总数取模，结果是一个确定的分区编号；同一个 Key 的消息，永远会进入同一个分区。
  2. 没指定消息 Key：采用轮询策略，第一条去分区 0，第二条去分区 1，循环往复分配，但单条消息依然只会进入一个分区。

哪怕你自定义分区器，正常业务逻辑也只会返回一个分区编号。也就是说，一次 send() 调用，消息只会被路由到一个分区，写入一次。

③ 最容易踩的坑：别把「副本」当成「多个分区」

很多人会产生 “一条消息在多个分区” 的错觉，本质是混淆了分区和副本这两个完全不同的概念：

  • 分区：是不同的数据分片，每个分区存的内容不一样，所有分区合在一起，才是这个 Topic 的完整数据。
  • 副本：是同一个分区的备份，内容完全一致，放在不同的 Broker 节点上，唯一作用是做数据冗余、保证高可用。

举个具体例子：一个 Topic 有 3 个分区，每个分区设置 3 个副本。

  • 一条订单消息写入了「分区 0」，它就只属于分区 0 这一个分片；
  • 但分区 0 有 3 个副本，分别存放在 3 台不同的 Broker 上，所以这条消息会在 3 台机器上各存一份备份；
  • 这 3 份备份属于「同一个分区的不同副本」，绝对不是「不同的分区」。

既然消息只存在一个分区里，为什么库存服务、短信服务、积分服务，都能收到同一条下单消息？

结论：和分区无关，和消费者组有关。

不同的业务服务，属于不同的消费者组。不同消费者组各自独立维护自己的消费进度（Offset），互不干扰，它们可以读取同一个分区里的同一条消息，各自处理各自的业务。

但这条消息本身，在磁盘的分区里只存了一份，并没有复制多份放到不同分区里。

一个分区能被多少个消费者组订阅？

结论：一个分区，可以被无数个消费者组同时订阅消费，没有数量限制。

不同消费者组之间完全独立、互不干扰，各自维护自己的消费进度（Offset），就像多个人用微信读书看同一本书，各自用各自的书签，互不影响。

结构示意图：

大白话解释：

  • 分区里的消息在磁盘上只存一份，不会因为订阅的组多就复制多份。
  • 每个消费者组有自己独立的 Offset 书签：库存组读到第 3 条，短信组读到第 5 条，积分组才读到第 1 条，互相完全不影响。
  • 这也是 Kafka 最经典的「发布 - 订阅模式」：一份数据，多个业务方各自消费，非常适合微服务场景下的多系统通知。

分区跟消费者组到底是什么关系？

核心记住两句话：

  1. 跨组：共享分区，独立消费（上面讲的，多个组可以读同一个分区）
  2. 组内：分区互斥，一对一分配（同一个消费者组里，一个分区只能分给一个消费单元）

同一个消费者组内，一个分区有且只能被一个消费线程（Consumer 实例）消费。

反过来，一个消费线程可以同时负责多个分区。

这个规则是 Kafka 保证「分区内消息不被重复消费、顺序消费」的核心。

我们用最经典的「3 分区 + 消费者组」场景举例，分三种情况看分配结果：

情况 1：组内线程数 < 分区数 → 一个线程负责多个分区

此时消费能力最弱，所有消息都靠一个线程处理。

情况 2：组内线程数 = 分区数 → 一一对应，性能最优

这是生产环境的理想状态：每个线程只负责一个分区，并发度最高，没有资源浪费。

情况 3：组内线程数 > 分区数 → 多余线程空闲浪费

划重点：消费者组的消费并发上限，等于 Topic 的分区总数。线程数超过分区数没有任何意义，只会白白浪费资源。

消费者组 vs 消费线程，是什么关系？

消费者组（Consumer Group）

纯逻辑概念，靠 group.id 标识。只要配置了同一个组 ID，不管在多少台机器、多少个进程里，都属于同一个消费者组，共同分担分区消费。

消费者实例（KafkaConsumer 对象）

真正干活的物理单元，就是代码里 new KafkaConsumer() 创建出来的对象。

KafkaConsumer 是非线程安全的，一个实例只能在一个线程里运行，不能多线程共享。

消费线程

运行消费者实例的线程。本质上：一个消费线程 = 一个独立的 KafkaConsumer 实例。

你可以在同一个进程里开 N 个线程，每个线程 new 一个 Consumer，它们都属于同一个消费者组；也可以部署在多台机器上，每台机器跑几个线程，共同组成一个大的消费者组。

结论：

  • 一个消费者组内可以有任意多个消费线程，它们可以分布在不同机器、不同进程里，只要 group.id 一致就行。
  • 一个消费线程对应一个 KafkaConsumer 实例，一对一绑定，不能共享。
  • 对 Kafka 服务端来说，它不关心你是多进程还是多线程，只认「组内有多少个 Consumer 实例」，按实例数量均匀分配分区。

代码对应关系

  • 每执行一次 new OrderConsumerThread()，就创建一个 KafkaConsumer 实例，运行在一个独立线程里。
  • 所有实例都配置了相同的 group.id = order-group，因此它们属于同一个消费者组。
  • Kafka 会自动把 order_topic 的分区，均匀分配给这 3 个线程对应的实例。

常见误区：

  1. 误区：一个 Consumer 实例开多线程并发消费，能提升速度
    1. 纠正：KafkaConsumer 是非线程安全的，多线程共享同一个实例直接报错。要并发消费，必须每个线程一个独立的 Consumer 实例。
  2. 误区：消费线程开得越多，消费速度越快
    1. 纠正：有效并发数上限等于分区数。超过分区数的线程只会空闲，不会提升消费速度。想提速，先加分区，再加线程。
  3. 误区：消费者组只能在一台机器上
    1. 纠正：消费者组是纯逻辑概念，成员可以跨机器、跨进程分布。这也是 Kafka 支持水平扩容的核心 —— 消费能力不够，就多加机器、多加实例。

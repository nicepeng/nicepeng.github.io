03 | 五分钟读懂 Kafka 整体架构：Broker、Topic、Partition、Producer、Consumer

摘要：在前两篇文章中，我们搞懂了 Kafka 的定位以及它与其他 MQ 的区别。这篇我们将正式进入 Kafka 的内部世界。很多人觉得 Kafka 的术语太多、架构太绕，其实只要掌握了它的 “物理、逻辑、客户端” 三层架构模型，一切都会豁然开朗。今天，我们就用”开一家超级物流公司”的类比，把 Kafka 的核心组件拆解得明明白白。

一、宏观视角：Kafka 的三层架构模型

要理解 Kafka，绝对不能死记硬背概念。我们可以把 Kafka 的整体架构分为三个维度来看：

接下来，我们逐层击破。

二、物理层：Kafka 的”基础设施”

想象你开了一家名为”Kafka”的超级物流公司，物理层就是你建设的硬件基础设施。

2.1 Broker（代理/节点）：

通俗解释：一台运行了 Kafka 服务的物理机或虚拟机。它是真正干活（接收、存储、转发数据）的地方。

2.2 Cluster（集群）

通俗解释：由多个 Broker 组成的集合。对外表现为一个统一的服务。

2.3 Controller（控制器）

通俗解释：在 Kafka 4.x 的 KRaft 模式下，集群中会选举出一个”Controller”节点。它不直接存储业务数据，而是负责管理元数据（比如哪个 Partition 在哪个 Broker 上、谁是 Leader 等）和协调集群状态。

💡 物理层小结：你部署了 3 台服务器运行 Kafka，你就有了一个包含 3 个 Broker 的 Cluster，其中 1 个 Broker 会被选为 Controller。

三、逻辑层：Kafka 的”数据组织方式”（核心重点！）

基础设施建好了，接下来要规定货物（数据）怎么分类、怎么存放。这是 Kafka 最精妙的设计，也是面试最爱考的地方。

3.1 Topic（主题）

通俗解释：Kafka 中数据的一级分类。生产者把消息发到指定的 Topic，消费者从指定的 Topic 订阅消息。

3.2 Partition（分区）

同一个 Topic 的分区，并不是都放在同一个 Broker 上的，而是会均匀分散在集群的不同 Broker 节点上。

这是 Kafka 分布式架构的核心设计：数据分片存储，每台机器只存一部分，既突破了单机容量限制，又能让多台机器同时提供读写服务，整体吞吐量翻倍。

  • 通俗解释：一个 Topic 的数据量可能极大，单台机器存不下、处理不过来。所以 Kafka 把一个 Topic 物理拆分成多个 Partition，分散到不同的 Broker 上。
  • 类比：”生鲜区”（Topic）太大了，我们把它拆分成 1 号冷库（Partition 0）、2 号冷库（Partition 1）、3 号冷库（Partition 2），分别建在北京、上海、广州的集散中心（Broker）。
  • 核心价值：
    • 高并发：生产者可以往不同的 Partition 并行写，消费者可以从不同的 Partition 并行读。
    • 水平扩展：Partition 的数量决定了系统并发能力的上限。

3.3 Replica（副本）

  • 通俗解释：为了防止某台机器突然断电或硬盘损坏，Kafka 会为每个 Partition 创建多个副本（Replica），存放在不同的 Broker 上。
  • 类比：北京 1 号冷库里的货，在天津也建一个备用冷库存一份一模一样的。北京停电了，天津立刻顶上。
  • Leader 与 Follower：
    • Leader（主副本）：负责处理所有的读写请求。
    • Follower（从副本）：不处理客户端请求，只负责默默地从 Leader 同步数据。当 Leader 挂了，Follower 会被选举为新的 Leader。

3.4 Offset（偏移量）

Kafka 的消息是持久化存储的，消费完不会立刻删除，那消费者怎么知道自己上次读到哪了？答案就是 Offset（偏移量）。

你可以把 Offset 理解成「看书的书签」：

  • 分区里的每条消息都有一个唯一的 Offset 编号，从 0 开始递增，相当于书的页码。
  • 消费者每次读完一批消息，就会记录一下当前读到了第几页（提交 Offset）。
  • 就算服务重启，下次再拉取的时候，也会从上次记录的页码接着往后读，不会重复也不会漏掉。

有两个非常重要的细节：

  1. Offset 是按「消费者组 + 分区」维度维护的，不是按单个消费者，也不是按 Topic。
  2. 不同消费者组的 Offset 互不影响。同一个 Topic 的同一条消息，可以被不同消费者组各自消费，互不干扰。

什么意思呢？

假设有一个主题：i_public_basic_info，现在消费者设置的是GROUP3消费组来消费，分区为0分区，现在GROUP3消费组已经消费到的offset是150，如果你想从头开始消费，可以把GROUP3改成GROUP4，因为GROUP4从来没有消费过这个主题和分组，那GROUP4消费的起始位置就是offset=0.

这也是 Kafka 和传统消息队列很大的区别：消息不是 “消费即删除”，而是可以被反复消费，非常适合多业务方订阅同一份数据的场景。

四、客户端层：Kafka 的”收发机制”

数据存好了，接下来看谁发货、谁收货。

4.1 Producer（生产者）：发货人

生产者本质上是一个「消息打包发货中心」，它要解决三个问题：

  1. 格式统一：把业务里的 Java 对象、字符串等，转换成 Kafka 能识别的字节数据。
  2. 高效传输：把小消息攒成一批再发，减少网络请求次数，提升吞吐量。
  3. 可靠送达：保证消息准确发到指定的 Topic 和分区，发送失败能重试。

4.1.1 一条消息的发送之旅（内部流程）

我们把生产者内部拆开，看看一条消息从产生到发出去，到底经历了什么：

第一步：构造消息，指定目标 Topic

业务代码里先确定消息要发到哪个 Topic（相当于快递要寄到哪个片区），同时可以选择指定消息 Key、分区号，不指定就用默认规则。

第二步：序列化器（Serializer）

Kafka 只认字节数据，不认 Java 对象、字符串。序列化器的作用，就是把我们的业务数据转换成字节数组，相当于把货物装进标准快递盒，方便运输和存储。

默认提供了字符串、整数等常用序列化器，复杂对象可以用 JSON、Avro 等自定义序列化方案。

第三步：分区器（Partitioner）

消息打包好之后，要决定发到这个 Topic 的哪个分区上，这就是分区器的工作，相当于快递分拣员，决定把包裹放到哪个货架上。

默认的分区规则很简单：

  • 如果指定了消息 Key：对 Key 做哈希计算，对分区数取模，同一个 Key 的消息一定会进到同一个分区，保证顺序。
  • 如果没指定 Key：默认轮询分发，这条去分区 0，下条去分区 1，均匀分配，保证负载均衡。

第四步：消息累加器（Accumulator）

这是 Kafka 高性能的关键设计之一。消息不是来一条发一条，而是先放到累加器里攒着，攒够一批再统一发出去，相当于快递站先攒够一整车货再发车，比一件件送效率高得多。

攒批有两个触发条件：

  • 大小阈值：批次大小达到 batch.size 默认值（16KB）就发。
  • 时间阈值：哪怕没攒够大小，等待时间到了 linger.ms 默认值（0ms，也就是不等待）也发。

这两个参数是后续性能调优的重点，我们调优篇再细讲。

第五步：Sender 线程发送

Sender 是生产者后台的一个独立线程，专门负责把累加器里攒好的消息批次，通过网络发送到对应的 Broker 节点上，相当于专门的货车司机，负责把货拉到中转站。

第六步：接收 Broker 确认响应

消息发到 Broker 后，Broker 会返回一个确认响应（ACK），告诉生产者 “消息我收到了”。生产者根据配置的确认级别，判断这次发送是否成功，失败了就自动重试。

4.1.2 三种常见的发送方式

根据对可靠性的要求不同，生产者有三种发送模式，对应不同的性能和可靠性：

  • 发后即忘（Fire-and-Forget）
    • 发出去就不管了，不关心有没有成功。性能最高，但可能丢消息，适合对可靠性要求极低的场景，比如不重要的日志。
  • 同步发送（Sync）
    • 发一条等一条的响应，成功了再发下一条。可靠性高，但是性能差，吞吐量低，适合对数据一致性要求极高的场景。
  • 异步发送（Async）
    • 发出去之后不用等，继续发下一条，Broker 响应回来之后通过回调函数通知结果。兼顾性能和可靠性，是生产环境最常用的方式。

4.2 Consumer（消费者）：收货人

消费者是离业务最近的角色，负责从 Kafka 里拉取消息并处理业务逻辑。这里有一个绝大多数新手都会搞错的点：Kafka 的消费者是主动拉消息，而不是 Broker 主动推消息。

4.2.1 核心模式：主动拉取（Pull） vs 被动推送（Push）

我们先搞懂两种模式的区别：

  • 推模式（Push）：Broker 主动把消息发给消费者，消费者被动接收。就像外卖员主动给你送餐。
  • 拉模式（Pull）：消费者主动定期去 Broker 那里问 “有没有新消息”，有就拉回来处理。就像你自己去快递驿站取件。

Kafka 为什么选择拉模式？核心原因是消费者能自己控制消费节奏。

  • 如果用推模式，Broker 不知道消费者的处理能力，发快了消费者处理不过来，直接被压垮；发慢了又浪费性能。
  • 用拉模式，消费者能根据自己的处理能力，决定每次拉多少、拉多快。处理得快就多拉，处理不过来就少拉，完全自己掌控，不会被流量冲垮。

打个比方：拉模式就像吃自助餐，你能吃多少拿多少；推模式就像服务员不停往你盘子里夹菜，不管你吃不吃得完，很容易撑到。

4.3 Consumer Group（消费者组）：消费能力翻倍的秘密

如果一个 Topic 消息量特别大，一个消费者处理不过来怎么办？答案就是用消费者组（Consumer Group）。

多个消费者可以组成一个消费者组，共同消费一个 Topic 的消息，分工合作，提升消费速度。就像一个快递站点，一个快递员送不完，就招几个快递员组成一个配送组，每人负责一片区域。

核心规则（重中之重）

一个分区，只能被同一个消费者组里的一个消费者消费。

但是，一个 Partition 可以被不同 Consumer Group 的 Consumer 消费。

这句话是理解消费者组的核心，我们用一张图直观展示：

这个规则带来了两个关键结论：

  1. 消费者数量 ≤ 分区数量：如果组里消费者数量比分区多，多出来的消费者会空闲，白白浪费资源。比如 3 个分区，4 个消费者，就有 1 个消费者没事干。
  2. 消费能力上限由分区数决定：想提升消费速度，光加消费者没用，必须先增加分区数。分区数是 Kafka 并发能力的天花板。

不同的业务场景，可以设置不同的消费者组。比如同一个订单 Topic，库存服务一个消费者组，短信服务一个消费者组，两者互不影响，各自消费全量消息，这就是典型的发布订阅模式。

单播和多播

Consumer Group 可以说是 Kafka 最精妙的设计，它用一套模型同时实现了单播（队列模式） 和多播（发布-订阅模式）。

场景 A：实现”单播”（负载均衡 / 队列模式）

假设 Topic 有 3 个 Partition，我们启动 1 个 Consumer Group，里面有 3 个 Consumer。

结果：3 个消费者平摊了工作，每条消息只被处理一次。这就是单播/队列模式。

场景 B：实现”多播”（广播 / 发布-订阅模式）

假设 Topic 有 3 个 Partition，我们启动 3 个不同的 Consumer Group（比如订单组、积分组、通知组），每个组里各有 1 个 Consumer。

结果：每条消息被所有组都消费了一次。这就是多播/发布-订阅模式。

场景 C：消费者数量 > 分区数量（浪费资源！）

假设 Topic 只有 2 个 Partition，但 Group 里有 3 个 Consumer。

结果：Consumer 3 永远分不到数据，白白浪费资源。

💡 避坑指南：在部署消费者时，Consumer 的数量绝对不要大于 Topic 的 Partition 数量！

七、思考题🤔

第一题

题干

场景设计题：你的系统有一个 Topic 叫 user-login-events，目前设置了 3 个 Partition。现在业务量暴增，你决定把 Partition 数量扩容到 6 个。

  1. 问题：扩容后，原本依赖 User_ID 进行 Hash 路由的消息，还能保证同一个用户的登录事件都进入同一个 Partition 吗？为什么？（提示：思考 Hash 取模算法）。

解析

这题其实是在问：分区扩容后，基于 Key 的路由还能保证同一个用户的数据在同一个 Partition 吗？

✅ 结论先行：绝对不能！这是一个非常经典的“生产事故”诱因。

很多团队在业务量暴增时，第一反应就是去增加 Topic 的 Partition 数量。但他们往往忽略了一个致命的问题：Kafka 默认的分区路由算法是基于取模（Modulo）的，一旦分区数改变，同一个 Key 的路由结果就会发生错乱。

🔍 底层原理剖析：Hash 取模算法

当 Producer 发送带有 Key 的消息时，Kafka 默认的分区策略（DefaultPartitioner）会执行以下计算：

核心公式：目标分区 = Hash(Key) % 分区总数

场景推演：扩容前后的“灾难”

假设我们有两个用户：User_A 和 User_B。

  • User_A 的 Hash 值是 10
  • User_B 的 Hash 值是 14

【扩容前：3 个 Partition】

  • User_A 的路由：10 % 3 = 1 👉 进入 Partition 1
  • User_B 的路由：14 % 3 = 2 👉 进入 Partition 2

此时，User_A 的所有登录事件都乖乖地待在 Partition 1 里，下游消费者可以完美保证 User_A 事件的顺序性。

【扩容后：6 个 Partition】

你执行了扩容命令，把 Partition 数量从 3 增加到了 6。

  • User_A 的路由：10 % 6 = 4 👉 进入 Partition 4 ❌
  • User_B 的路由：14 % 6 = 2 👉 进入 Partition 2 ✅ (碰巧没变)

灾难发生了： User_A 的历史登录事件还在 Partition 1，但扩容后新产生的登录事件却跑到了 Partition 4！ 如果你的下游业务（如 Flink 实时计算、或者依赖单分区顺序性的状态机）强依赖“同一个用户的数据必须在同一个 Partition”，那么数据乱序、状态计算错误等严重 Bug 就会接踵而至。

🛠️ 破局之道：如何优雅地解决分区扩容问题？

既然直接扩容会破坏 Key 路由，那我们该怎么办？以下是三种常见的解决方案：

方案一：未雨绸缪（最佳实践 ⭐⭐⭐⭐⭐）

“不要等渴了才挖井。”

在创建 Topic 之初，就根据未来 1~2 年的业务增长预估，一次性设置足够多的 Partition。

  • 技巧：分区数通常设置为 12、24、36、48 等数字。
  • 为什么？ 这些数字有很多公约数。如果未来你需要调整消费者数量（比如从 12 个增加到 24 个），这些数字都能被完美整除，避免消费者闲置。同时，前期多建几个空的 Partition，对 Kafka 的性能几乎没有影响。
方案二：新建 Topic + 数据双写/迁移（最稳妥 ⭐⭐⭐⭐）

如果历史包袱太重，绝对不能破坏原有路由，那就新建一个 Topic。

代价：需要修改业务代码，且短期内会有两套 Topic 并存。

方案三：自定义 Partitioner（一致性哈希）（不推荐 ⭐⭐）

理论上，你可以自己写一个实现 Partitioner 接口的类，使用一致性哈希（Consistent Hashing） 算法来替代取模算法。

  • 优点：扩容时，只有少部分 Key 的路由会发生迁移，大部分 Key 保持不变。
  • 缺点：
    • 开发和维护成本极高。
    • 一致性哈希极易导致数据倾斜（Data Skew），即某几个 Partition 数据量暴增，其他 Partition 空空如也，违背了 Kafka 负载均衡的初衷。
    • 所有的 Producer 端都必须统一替换自定义的 Partitioner 类，漏掉一个就会导致数据错乱。

💡 架构师箴言： “Kafka 的 Partition 扩容是一个破坏性操作（针对 Key 路由）。在架构设计时，宁可前期多建分区，也绝不后期盲目扩容。”

第二题

题干

故障排查题：你的消费者组里有 4 个 Consumer 实例，但是监控显示只有 3 个实例在疯狂干活，第 4 个实例的 CPU 和内存占用几乎为 0。请结合本篇知识，推测可能的原因是什么？

解析

✅ 结论先行：90% 的概率是因为 Topic 的 Partition 数量小于 Consumer 数量。

这是 Kafka 初学者和 even 一些老手都容易踩的坑。要排查这个问题，我们需要深刻理解 Kafka 消费组的分配铁律。

文章中我们强调过：

一个 Partition 在同一时刻，只能被同一个 Consumer Group 内的 一 个 Consumer 消费！

嫌疑 1：Partition 数量 < Consumer 数量（最常见 ⭐⭐⭐⭐⭐）

场景还原：

你的 Topic user-login-events 只有 3 个 Partition，但你觉得“为了高并发，我启动 4 个 Consumer 实例吧”。

分配结果：

验证方法：

使用 Kafka 命令行工具查看 Topic 的 Partition 数量和消费组的分配情况：

如果在输出结果中，看到某个 Consumer 的 CURRENT-OFFSET 和 LOG-END-OFFSET 都是 -（空），说明它确实没分到 Partition。

解决办法：

  • 增加 Partition：将 Topic 的 Partition 数量扩容到 4 个或更多（注意思考题一的坑！）。
  • 减少 Consumer：关掉 1 个 Consumer 实例，保持 Consumer 数量 = Partition 数量。

嫌疑 2：正在发生 Rebalance（消费者重平衡）（较常见 ⭐⭐⭐⭐）

场景还原：

Kafka 消费组在以下情况会触发 Rebalance（重新分配 Partition）：

  1. 有新的 Consumer 加入群组。
  2. 有 Consumer 崩溃或主动离开。
  3. Consumer 处理消息太慢，超过了 max.poll.interval.ms（默认 5 分钟），被 Coordinator 踢出群组。
  4. Topic 的 Partition 数量发生变化。

在 Rebalance 期间，整个消费组会暂停消费（Stop The World）。如果 Rebalance 过程非常缓慢，或者发生了“Rebalance 风暴”（不断触发），某些 Consumer 可能会长时间处于未分配状态。

验证方法：

查看 Consumer 的应用日志，搜索关键字 Revoking、Partitions assigned、Rebalance。如果日志里疯狂刷这些词，说明陷入了 Rebalance 风暴。

解决办法：

  • 检查消费者代码，确保 poll() 之间的处理逻辑不要耗时过长。
  • 适当调大 max.poll.interval.ms 和 session.timeout.ms。
  • 在 Kafka 2.4+ 版本中，开启增量式 Rebalance（CooperativeStickyAssignor），避免全局 Stop The World。

嫌疑 3：网络分区或心跳超时（偶发 ⭐⭐⭐）

场景还原：

Consumer 4 所在的机器网络出现抖动，导致它无法按时向 Group Coordinator 发送心跳（Heartbeat）。Coordinator 认为 Consumer 4 已经“死了”，于是把它的 Partition 分配给了别人。但 Consumer 4 的进程其实还活着，只是处于一种“被孤立”的假死状态。

验证方法：

检查 Consumer 4 所在机器的网络状况，以及日志中是否有 Heartbeat session expired 或 Coordinator not available 的报错。

解决办法：

  • 重启 Consumer 4 进程。
  • 检查网络链路，优化 heartbeat.interval.ms 配置。

嫌疑 4：配置错误（低级失误 ⭐⭐）

场景还原：

  • group.id 写错了：Consumer 4 的 group.id 配置成了另一个组，导致它其实不属于这个 4 人小队。但这通常会导致它去独立消费所有 Partition，而不是闲置。
  • 订阅了错误的 Topic：Consumer 4 代码里 subscribe 的 Topic 名字拼错了，订阅了一个不存在的 Topic，自然拉不到数据。

验证方法：

仔细核对 Consumer 4 的配置文件或启动参数。
